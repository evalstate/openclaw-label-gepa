# Decision Procedure

Route by the intended maintainer-owned behavior in the title/body. Use comments, labels, files, diffs, and tests only to confirm ownership. A topic qualifies only when its contract, state, lifecycle, policy, delivery path, or user behavior changes independently. Treat bug words as symptoms; label the repaired surface, not hosts, helpers, examples, paths, or consequences.

# Cardinality Rules

Default to 1-2 topics; use 3 for separate central surfaces; use 4-5 only for truly cross-cutting protocol/auth/security/runtime work. Include all clearly qualifying topics, but each extra must pass: would that maintainer need review if the other labels were already correct?

Final prune: drop labels supported only by a path, fixture, build knob, generic fix/fail word, example provider/platform, config key, session/context mention, "message/tool" wording, or downstream effect. Prefer the precise owner over a broad symptom label.

# Boundary Overlays

- Add `reliability` only for runtime failure-handling: retries/timeouts/backoff, crashes/leaks, cleanup, caps/TTL on live state, stuck/terminal state, data loss, races, overload, recovery, or durable failure reconciliation. Suppress it for ordinary bugs, stale/wrong counters or status, empty/misrendered payloads, provider/API formatting compatibility, missing chat/notification text, CI/test/build speed, or fixes wholly owned elsewhere.
- Add `notifications` for outbound notify/reply/completion/ack/announcement behavior, delivery gates, sent-message handling, or delivery-failure reconciliation. Named chat delivery or acknowledgement changes usually need `chat_integrations` plus `notifications`; add `reliability` only when recovery/stuck-state handling also changes.
- Add `telemetry_usage` for token/usage/cost counts, metrics, traces, diagnostics, status, freshness, and reporting. Prefer it over `reliability` for stale or incorrect counters/status; pair `ui_tui` only when the displayed UI changes.
- Add `chat_integrations` only for a named chat platform, channel adapter, ingestion, or chat delivery surface. Suppress it for generic assistant-message UI, UI copy/rendering, non-chat command output, or internal notifications without a chat adapter.
- Add `ui_tui` for OpenClaw UI/TUI/dashboard/mobile/display controls and visible status. Do not add it for menus or commands that live solely inside a chat platform/config, or for API/telemetry fields not shown to users.
- Add `coding_agents` for external or OpenClaw-managed coding-agent/subagent integrations or flows, including Codex/Claude Code/Gemini/Pi-style agents. Do not replace it with `gateway`, `sessions`, or `agent_runtime` when those merely host the flow; label ACP protocol/delivery separately.
- For ACP work, include `acp` for protocol, nodes, sessions, binding, parent/child, follow-up, output, or delivery semantics; include `acpx` for ACPX transport/proxy/worker/managed backend/binding. Wrong or empty ACP outputs are `acp`/`acpx`, not `reliability`, unless recovery logic is added.
- Add `sessions` only for session identity, lifecycle, persistence, binding, transcript, resume/reset, parent/child state, stores, or cleanup. Suppress it for ordinary context, terminal artifacts, notification paths, or files that merely mention sessions.
- Add `gateway` only for gateway routes/protocol, daemon startup/restart/health, gateway-owned execution, or gateway state. Suppress it when the gateway is only the host/path for chat config, subagents, notifications, provider proxying, or API handlers owned elsewhere.
- Add `tool_calling` only for model tool-call protocol/deltas, function schemas, tool-result transcript/routing, or invocation parameter coercion. Suppress it for command output, slash/chat acknowledgements, provider text/thinking/content blocks, TTS, vision/screenshots, browser output, prompt hints, and config-like options.
- Pending permissions, approval prompts, decisions, modes, and pending-approval state are `approvals`; TTL/caps/close-clear for that state add `reliability`. Do not use `memory` for pending approvals, transcripts, context, or generic remembered state; use it only when memory/index/vector/provider behavior changes.
- Add `config` only for operator-facing settings, persisted schema/defaults, validation/repair, env/options, policy/allow-deny, or migration. Do not add it for internal constants, test/build knobs, example config keys, or unsupported options passed as another surface's compatibility bug.
- Use `tests_ci` when tests, CI, fixtures, smoke tests, or platform test infrastructure are the subject; pair `packaging_deployment` for install/build/image/release artifacts. Do not add `config` or `reliability` for test-only speed or smoke-build changes.
- Provider API rendering, streaming, usage chunks, TTS/vision/embeddings, auth/routing, catalogs, or compatibility is `inference_api`. Add `self_hosted_inference` only for engine setup, preflight, lifecycle, backend compatibility, crashes/timeouts, or local/self-hosted backend behavior; suppress it when such an engine is only an example of generic API rendering.
- Add `codex` only for Codex runtime/auth/ACP/plugin/command compatibility, not cosmetic app/UI changes. Add `auth_identity` for auth bridges, account binding, credential propagation, profile/scope selection, or token-only auth. Add `security` for credential boundaries, isolated homes, token scope, secret exposure, permissions/access control, sandbox escape, or private access.
- Use `hooks`, `api_surface`, and `skills_plugins` only when those product contracts themselves change.

# Suppression Rules

Suppress labels inferred only from paths, package names, tests, examples, platform names, generic failure wording, config keys that merely implement another feature, or session/context/message/tool words that do not change those contracts.

Before final output, remove weak extras until each remaining topic names an independently reviewable maintainer surface.
