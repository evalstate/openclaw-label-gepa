```md
# Decision Procedure

Read the title and main problem/feature statement first; use body, comments,
labels, changed files, and diff only to confirm or disambiguate central
interests. Select every allowed topic that is a central maintainer-owned
concern under the fixed taxonomy and boundary overlay.

# Cardinality Rules

Include every topic whose inclusion rule in the fixed overlay is satisfied; do
not drop a qualifying topic to keep the output short. Use 1-3 topics by
default, 4-5 only when genuinely cross-cutting, never more than 5. Drop labels
that are only symptom locations, mechanisms, paths, examples, or consequences.
Prefer replacing a vague failure label with the surface that owns the fix.

# Boundary Overlays

- Add `telemetry_usage` whenever the central user-visible or API-visible change is usage/accounting/status/diagnostic data such as token counts, usage chunks, costs, traces, metrics, or freshness of those displays; do not recast stale/incorrect counts as `reliability`.
- Add `notifications` when outbound delivery, acknowledgements, announcements, completion/result delivery, delivery gates, or chat reply payload/path behavior is central. For named chat surfaces, co-label `chat_integrations` plus `notifications` when delivery semantics change.
- Add `coding_agents` for integrations or orchestration involving external coding agents or OpenClaw-managed agent products; do not downgrade these to `gateway`, `agent_runtime`, or `sessions` just because the implementation uses those internals.
- For ACP/ACPX work, label `acp` for protocol/session/binding/parent-child/delivery semantics and `acpx` for ACPX transport/proxy/worker/binding artifacts; include both when both layers are central.
- Add `sessions` only for session identity, lifecycle, persistence, binding, transcript, resume/reset/cleanup, parent/child session state, or thread-bound follow-up state. Do not add it for a mere session context mention or UI/status data stored in a session.
- Add `approvals` for approval prompts, permission decisions, approval modes, pending approval state, or bounding/expiring/clearing pending approvals, even when the bug also involves MCP state or reliability.
- Add `hooks` when emitting, filtering, prioritizing, registering, or securing hook events is a product requirement; a missing hook emission is not merely reliability.
- Add `tests_ci` with `packaging_deployment` when the central deliverable is install/build smoke tests, CI fixtures, coverage, or test-environment performance for packaging artifacts.
- Add `self_hosted_inference` with local-model preflight or retries when the concern is integration with Ollama/LM Studio/llama.cpp/vLLM-style engines, even if the title says local model.
- Add `security` with auth/identity/codex/acp work when credential isolation, auth bridging, token scope, private access, sandbox escape, or secret exposure is a central boundary.
- Add `config` only for operator-facing options, schemas, persisted shape, validation/repair, defaults, env settings, policies, migrations, or allow/deny settings; an internal flag or build/test knob is not enough.
- Add `api_surface` only when an external CLI/HTTP/SDK/webhook/SSE contract or request/response shape changes; chat slash-command acknowledgements and platform delivery behavior usually route to `chat_integrations`/`notifications`.

# Suppression Rules

- Do not add `reliability` for a generic bug/fix, stale display, missing UI rendering, wrong API compatibility detail, CI-only failure, or a failure that merely motivates a change owned by another surface.
- Add `reliability` only when retries/recovery, timeout/crash/leak, stuck state, data loss, lifecycle cleanup, overload, race, caps, or operational failure-mode policy is itself central.
- Do not add `tool_calling` unless the central change is model tool-call protocol, schemas, deltas, function-call arguments, tool-result transcript handling, or tool-result routing; generic command output, ACP node output, thinking/rendering blocks, screenshots, TTS, or config options are insufficient.
- Do not add `chat_integrations` for generic message UI or assistant-message controls without a named chat platform, channel adapter, ingestion path, or chat delivery surface.
- Do not add `codex` just because a Codex-branded package/app/file is touched; require Codex runtime, auth, ACP, plugin, command compatibility, or integration behavior.
- Do not add `memory` for generic remembering, pending state, transcript/session context, or cache lifetime unless the memory/indexing/vector-store surface is central.
- Do not add `agent_runtime` for ACP parent/child delivery, external coding-agent integration, or subagent product behavior unless runtime startup/loop/backend/model-call orchestration is the central concern.
- When a row looks over-broad, remove labels supported only by files, tests, examples, implementation mechanisms, or downstream consequences.
```
