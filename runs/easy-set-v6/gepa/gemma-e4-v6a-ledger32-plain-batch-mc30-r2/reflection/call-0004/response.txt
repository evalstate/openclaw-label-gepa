# Decision Procedure

Read title and main change first; use body, comments, labels, files, and diff only to confirm ownership. Select every allowed topic whose maintainer must review the changed behavior, contract, state, lifecycle, policy, delivery path, or user-visible surface.

Prefer the changed product surface over symptoms, paths, helper code, examples, or bug words. A co-label qualifies only if that topic has an independent behavior or contract change, not merely because it is mentioned or hosts the implementation.

# Cardinality Rules

Use 1-3 topics by default; use 4-5 only when several surfaces independently change. Never output more than 5.

Before final output, remove labels supported only by generic fix/fail wording, file paths, tests, build knobs, platform examples, config key names, incidental session/context mentions, generic message/tool wording, or consequences of another surface.

If a row looks like "surface A implements delivery/status/config for surface B", include both only when both are operator- or user-facing contracts; otherwise keep the owner surface.

# Boundary Overlays

- `reliability`: add only for runtime failure-handling or operational safety: retries, timeouts, crashes, leaks, caps/TTL/cleanup for live state, stuck/terminal states, races, overload, recovery, data/result loss, failed delivery recovery, or durable send/run failure. Suppress for ordinary bugs, stale/wrong display values, missing/empty rendered text, provider formatting/compatibility, unsupported option cleanup, CI/test-only failures, speed work, or notification/API/UI corrections without recovery mechanics.
- `telemetry_usage`: use for token/usage/cost counts, metrics, traces, diagnostics, status, freshness, or reporting, including when shown in UI or sessions. Prefer this over `reliability` for stale or incorrect counters/status.
- `notifications`: use for outbound notifications, replies, acknowledgements, completion/announce delivery, delivery gates, sent-message handling, or delivery payload/path changes. Chat delivery changes often need both `chat_integrations` and `notifications`; do not substitute `reliability` unless failure recovery/loss prevention is also central.
- `chat_integrations`: use only for a named chat platform, channel adapter, chat ingestion, or chat delivery surface. Suppress for generic assistant messages, UI message rendering/copy buttons, or non-chat command/message actions.
- `coding_agents`: use for external or OpenClaw-managed coding-agent integrations/flows, including subagent-style flows and Codex/Claude Code/Gemini/Pi integration. Do not replace it with `gateway`, `sessions`, or `agent_runtime` unless those surfaces independently change.
- ACP: include `acp` for ACP protocol/session/node/binding/parent-child/output/delivery semantics. Include `acpx` when ACPX transport/proxy/worker/backend/binding is central. Wrong/empty ACP outputs are `acp`/`acpx`, not `reliability`, unless recovery/state safety is added.
- `sessions`: use only for session lifecycle, identity, persistence, binding, transcript, resume/reset, parent-child state, stores, or cleanup. Suppress for incidental context/session files, terminal artifacts, notification paths, or subagent/gateway mentions.
- `gateway`: use only for gateway routes, protocol, state, startup/restart, service health, or gateway-owned execution. Suppress when gateway merely hosts chat config, subagents, notifications, provider proxying, or API handlers owned elsewhere.
- `tool_calling`: use only when the model tool-call protocol changes: tool-call deltas, schemas, function/tool arguments, tool result transcript/routing, or invocation parameter coercion. Suppress for shell/command output, slash command acks, chat sends, TTS, Responses/API content rendering, thinking/content blocks, screenshots, browser vision, and config-like options.
- `approvals`: use for permission prompts, approval decisions/modes, pending approval state, or approval TTL/caps/cleanup. Pending approval leaks are `approvals` plus `reliability`, not `memory`, unless memory indexing/search/provider behavior changes.
- `memory`: use only for memory indexing/search, embeddings/vector stores, provider state, archival/recovery, or active memory behavior. Do not use for generic remembered state, context, transcripts, pending queues, or approval state.
- `config`: use for operator-facing settings, persisted schema/defaults, loading/validation/repair, policy/allow-deny options, environment options, or migrations. Suppress for build/test knobs, internal constants, example config keys, or keys used only to implement another surface.
- `tests_ci`: use when tests, fixtures, CI, mocks, smoke tests, or platform test infrastructure are the subject. Pair with `packaging_deployment` for build/install/image smoke work; do not add `reliability` or `config` unless runtime/operator behavior also changes.
- Inference: use `inference_api` for provider/API rendering, streaming, TTS, vision, embeddings, usage chunks, catalogs, auth/routing, request/response compatibility. Add `self_hosted_inference` when engine setup/lifecycle/compatibility or local/self-hosted backend behavior is central; local-model preflight against such engines qualifies.
- `codex`: use only for Codex runtime/auth/ACP/plugin/command compatibility, not cosmetic app/UI changes. Add `auth_identity` for auth bridges, account binding, credential propagation, profile/scope selection, or token-only auth. Add `security` for credential boundaries, isolated homes, token scope, private access, permissions, secret exposure, sandbox escape, or access control.
- `ui_tui`: use for actual UI/TUI/dashboard/mobile display or interaction. Suppress for chat-platform menus, command internals, API/provider behavior, or telemetry fields not changed as user-facing UI.
- `api_surface`: use only for external API/CLI/HTTP/SDK contracts, request/response shapes, webhooks, SSE, or documented command contracts. Suppress for slash command acks, chat acknowledgements, UI actions, and provider internals.
- `hooks` and `skills_plugins`: add only when their own registration, loading, lifecycle, payload, filtering, SDK/runtime, SecretRefs, doctor/check, or security contract is central.

# Suppression Rules

- Suppress `reliability` for generic bug labels, compatibility/rendering fixes, stale metrics, missing notification payloads, test/build speed, or failures that merely motivate another surface.
- Suppress `tool_calling` unless the model tool-call contract itself changes.
- Suppress `agent_runtime` for ACP orchestration or coding-agent integration unless core startup, loop, backend, adapter, or model-call orchestration is the deliverable.
- Suppress `chat_integrations`, `ui_tui`, `sessions`, `config`, and `gateway` when they are only incidental hosts, paths, examples, labels, or implementation mechanisms.
